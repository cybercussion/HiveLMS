Hive
Version 1.0 beta 2

Multi-device LMS

Hive was created as a light weight LMS(Learning Management Server) that supports mobile devices, and desktops. The goal of this project is to show that utilizing HTML5, Cascading Style Sheets (CSS), JavaScript(ECMAScript), Webkit, and other 3rd Party jQuery plugins, can accomplish a multi-device solution.

The LMS has been written in PHP and mySQL, with SCORM 2004 implemented. This is a work in progress. Recently this was created as part of a prototype, so SCORM 2004, gradebooks, forums or things of that sort are currently lacking.

Browser Compatibility

The scope of the project focused on modern browsers (FireFox, Safari/Mobile, Chrome/Mobile), but made the attempt to remain compliant with Internet Explorer 8. Some of the main difficulties with older browsers is they lack the JavaScript horsepower, transparent PNG support, shadowing, and open standards to make this an easy task. Opera currently does not support CSS gradients, and we've noticed some rounded corner issues. One of the main things to remember is this portion is typically cosmetic, and there are always work arounds.

Zero browser specific code has been written, but we do detect support of the webkit or touch.

Some iOS notes: The original goal was to allow this to be a Web App "Add to Homescreen", however a couple things made that difficult due to the way the LMS launches external content:

  1. IFRAME's are not well supported on the iOS (however, iOS 4.2 has made some updates to the IFRAME support. Those are currently being tested)

  2. New Windows override the Web App window, thus eliminating the LMS window. This terminates the SCORM communication and there is no way back.

Again, this only applies to adding the Hive LMS to your "homescreen", and for now the "Web App" capability was disabled. It's possible it may come back after the iOS 4.2 update.


Content Approach

In order to use as much as the browser real-estate as possible, the Shareable Content objects are allowed to scale to the full size of the browsers viewport. This was especially important on the iPad, and that inspiration crossed over to the desktop.

In order to keep both the iPad and iPhone/Android 2.2 devices in visual sync, we utilize the @media abilities in CSS to override the default behavior for devices at there maximum widths.

One very important thing after building HTML content - There are many, many ways to load and display content. This implementation attempted to address screen lag between pages, webkit transitions or jQuery margin based ones to mimic the same thing. Websites by nature load when they load, and the desire was to smooth that out. At the same time, preloader animations were undesirable. This approach allows for the content to load in the background, displaying the first page as soon as its available.

Tips/Tricks

The following items were discovered as possible ways to improve the design, flow, bandwidth and amount of HTTP calls (lag) the SCO (Shareable Content Object) makes:

  1. JavaScript Templates: This allows for a data model to markup synchronization, and a reduction of HTTP calls. It also allows for unique identifiers to be better maintained.

  2. CSS Sharing: Merge as much of the shared styling at the top level, as not to keep making it get downloaded per template/page. Possibly consider merging it all into one file.

  3. Minify/Merge: Shrink your code down, some times as much as 79% reduction, and limits HTTP traffic on final Production deployments. Removing useless formatting, whitespace, line returns etc ... This interface was reduced by 57%, then with gzipping a total of 78% was achieved. 15 HTTP calls were eliminated.

  4. Pre-render in background: Telling a page to transition in, and initialize at the same time will result in visual lag.

  5. Master the Webkit: If you go in trying to animate with JavaScript you're going to be shockingly unimpressed with the performance on mobile devices. Many things you had to do with a bunch of JavaScript, now is accomplished thru CSS. And unfortunately, if thats your first impression of HTML5, you probably browsed on. Never fear however, the webkit has your back!

  6. Touch vs Mouse: These events can be managed universally, but they need to be managed.

  7. iOS Fixed Position: position: fixed; doesn't work the way you're use to. If you have scrolling content, and a desire for a sticky header or footer, you have to consider using a Webkit scroll solution. These vary in size, and complexity.

  8. Target Delivery: Frameworks exist for mobile devices, but most those frameworks are geared towards Webkit browsers. You try to view these with Internet Explorer, FireFox or Opera and they simply will not look right. Planning has to go into what will be looking at your content, because you'll have to make design compromises.

  9. Audio: Mobile devices treat audio a bit different than Desktops. Audio can only be triggered on a user action, and that can also result in some lag attempting to trigger sounds. It was enough lag, that it was disabled for touch devices.

  10. Tools/Tricks YSlow from yahoo and Page Speed both great resources and add-ons to FireFox with FireBug. It will help you find the pork and deal with it accordingly.

It is a delicate balance, and requires scrutiny for the application of all the items above. Shared the facts about the build of this project (*see Nutrition Facts).
